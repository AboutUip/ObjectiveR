# ObjectiveR 执行模型规范

## 0. 术语定义

- **启动输入**：解释器接收的启动参数，可为文件路径或目录路径。
- **项目空间**：由唯一入口文件 `main.obr` 的语言空间唯一确定的资源边界。
- **项目根目录**：项目空间的根节点目录，用于资源扫描与路径规范化。
- **入口函数**：`main.obr` 中实现的 `deRfun main():void`（声明见 [`system.mr`](system.md) 与 [`moduleR.md` 第 7.13 节](moduleR.md)）。

## 1. 启动输入与入口文件约束

解释器允许以下两类启动输入：

- 直接指定 `main.obr` 文件；
- 指定项目目录（文件夹）。

统一入口约束如下：

- 入口文件名必须严格匹配 `main.obr`（大小写敏感）。
- 无论以文件方式启动还是以目录方式启动，项目空间内有且只能存在一个 `main.obr`。
- 若未找到 `main.obr`，必须报错并终止。
- 若检测到多个 `main.obr`，必须报错并终止。
- 非 `main.obr` 的 `.obr` 文件不得作为执行入口。
- 目录启动模式下，解释器必须先识别唯一 `main.obr`，再建立运行上下文。

## 2. 语言空间控制原则

- 除 `main.obr` 外，其他所有 `.obr` 代码文件都必须受 `main.obr` 的语言空间控制。
- 解释器在建立运行上下文时，先识别并锁定唯一 `main.obr`，再读取其预处理指令并确定项目根与扫描范围。
- 若 `main.obr` 的语言空间声明与实际文件路径无法建立一致映射，必须报错并终止。
- 项目空间的确定仅取决于 `main.obr` 的语言空间，与从哪个目录启动解释器无关。

## 3. 项目根目录判定

设 `M` 为入口文件 `main.obr` 的实际文件路径。

- 当 `main.obr` 的 `#LINK` 语言空间为 `/`（或未显式声明，按默认 `/` 处理）时：
  - 当前 `main.obr` 所在目录即项目根目录。
  - 解释器应以该根目录作为资源扫描起点。

- 当 `main.obr` 的 `#LINK` 指向 `/main/main.obr` 时：
  - 项目根目录应识别为入口文件上一层目录（即可将 `M` 视为项目内 `/main/main.obr` 对应实体）。
  - 解释器应以识别出的根目录作为资源扫描起点。

目录启动补充：

- 当启动输入为目录时，解释器必须先识别唯一 `main.obr`，再完全按其语言空间执行根目录判定与扫描。
- 启动目录仅用于定位候选入口，不得用于覆盖、收窄或扩展由 `main.obr` 语言空间确定的项目空间。

说明：

- 根目录判定必须在资源扫描前完成。
- 项目根目录一旦确定，后续规范化与访问判定必须使用同一根目录基准。

## 3.1 `libs/` 目录（解释器托管 `.mr`）

项目根目录确定后，对 **`libs/`** 的清单与存在性要求见 [系统头与 `libs/` 规范](system.md) 第 2 节。满足清单的前提下，实现可采用下列等价策略之一或组合：

- 启动前使 `libs/` 目录内容与 `system.md` 第 2.1 节清单一致（含「删除旧目录再写入」）。
- 运行结束后删除 `projectRoot/libs/`（下次运行再按上一步保证）。

行为与参考实现的对照见 [`docs/blink/modules.md`](../blink/modules.md)（`LibsProvisioner`）、[`docs/blink/pipeline.md`](../blink/pipeline.md)（`finally` 清理）。

## 4. 资源扫描与资源表构建

在根目录判定完成后，解释器必须执行全量资源扫描并构建资源表。

### 4.1 扫描范围

- 扫描范围为项目根目录及其子目录。
- 扫描对象包含：
  - 所有 `.obr` 代码文件
- 所有 `.mr`（moduleR）文件
  - 其他非 `.obr` 杂项资源（如配置、数据、静态资源等）

### 4.2 资源表要求

- 每个资源必须记录“相对 `main.obr` 的相对路径”。
- 资源表必须构建为树结构（目录节点 + 文件节点）。
- `main.obr` 本身必须被扫描并加入树结构，不得遗漏。
- `.mr` 文件必须被扫描、记录并在资源表中显式标识为 moduleR 类型节点。

### 4.3 一致性约束

- 相对路径计算规则必须全局唯一且稳定。
- 同一路径在资源表中只能对应一个节点。
- 构建完成后，后续模块解析、访问控制、加载行为都必须依赖同一资源表。
- 资源表中每个节点的路径必须可由项目根目录唯一推导，不得存在等价重复路径。

## 5. 执行入口与调用约束

- 解释器永远只执行 `main.obr` 中对 `deRfun main():void` 的实现作为程序执行入口（函数头由 [`system.mr`](system.md) 声明，见 [`moduleR.md` 第 7.13 节](moduleR.md)）。
- 除该入口调用外，其余所有函数调用都必须由 `main` 函数直接或间接拉起。
- 任何绕过 `main` 入口直接触发其他函数执行的行为均不符合规范。
- 若 `main.obr` 不存在可识别的 `deRfun main():void` 实现，必须报错并终止执行。

### 5.1 函数内 `static var`（静态存储）

- 由 `static` 确定的变量使用**函数级静态存储**（槽位跨调用保留，直至进程内解释器实例生命周期结束）。
- **每次**执行到 `static var[type] name …` 语句时：若当前函数的静态存储中**已存在**同名槽，必须报错（`E_RT_STATIC_LOCAL_REDECL`），视为对同一静态名的重复声明；**不得**采用「仅首次初始化」的 C/Java static 局部语义。
- 与 [`static.md`](static.md) 第 4 节（不可逆 / 不得重复标识）一致；与 [`visibility.md` 第 8 节](visibility.md) 正交说明一致。
