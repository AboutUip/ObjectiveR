# ObjectiveR `.mr`（moduleR）文件规范

## 1. 文件定义

- `.mr` 文件为 ObjectiveR 的 moduleR 文件。
- moduleR 文件用于承载函数头定义、宏定义等模块化声明内容。
- `.mr` 文件仅允许声明，不允许实现函数体。

## 2. 项目根目录关系

- `.mr` 文件所在项目空间必须由 `main.obr` 决定项目根目录。
- `.mr` 文件的归属与路径规范化必须使用与 `.obr` 文件相同的项目根基准。

## 3. 扫描与记录要求

- 解释器执行项目扫描时，必须同时扫描 `.obr` 与 `.mr` 文件。
- 扫描到的 `.mr` 文件必须写入资源表。
- 资源表中必须显式标识 `.mr` 文件类型（moduleR）。
- `.mr` 资源节点必须记录相对 `main.obr` 的相对路径，并纳入统一树结构。

## 4. 一致性约束

- `.mr` 文件的路径去重、节点唯一性规则应与 `.obr` 文件一致。
- 同一路径不得在资源表中出现多个 `.mr` 等价节点。

## 5. `import` 导入规则

### 5.1 语法

- `.obr` 文件中的 `import` 语句必须位于“最外层（顶层）”，即不得出现在任何嵌套结构中（例如函数体、方法体、类体、局部作用域等）。
- `import` 语句在文件内的文本位置顺序不影响语义；当解释器进入执行上下文时，必须扫描并处理该 `.obr` 的所有最外层 `import`。
- `.obr` 文件可以多次使用 `import xxx` 导入同一 `xxx.mr`，但必须满足幂等性：多次导入不得触发多次初始化或多实例绑定。
- `import` 目标名不包含 `.mr` 扩展名。
- 解释器必须将 `import xxx` 解析为对 `xxx.mr` 的导入请求。

示例：

```text
import test
```

该语句表示导入 `test.mr`。

### 5.2 解析时机

- 当某个 `.obr` 文件被激活到执行上下文中时，解释器必须扫描并读取该 `.obr` 所有最外层 `import` 语句，并对其中每一条 `import xxx`：
  - 解释器必须立刻解析对应 `xxx.mr` 文件；
  - 若该 `.mr` 已解析，必须复用已有解析结果，不得生成新的独立解析结果。
- 解析结果必须登记 `.mr` 中声明的函数信息、命名空间信息及宏信息。

**例外（系统头 `system.mr`）**：

- 即使源码中未出现 `import system`，解释器也必须在加载任意 `.obr` 时**等价于已导入** [`system.mr`](system.md)（隐式导入）；若源码中显式写了 `import system`，语义必须与隐式导入一致，不得产生第二份模块实例。

进一步约束：

- 除 `system.mr` 的隐式导入规则外，其它 `.mr` 的加载/解析仍须由显式 `import` 触发（见第 6.1 节）。
- 对同一 `.mr` 文件，若已存在解析结果，则后续相同导入应复用该结果，而不得生成新的独立解析结果。

### 5.3 可访问性

- 导入该 `.mr` 的 `.obr` 文件可访问由该 `.mr` 声明或标识的函数。
- 与模块名、路径相关的冲突判定见第 5.4 节。

### 5.4 模块名唯一性（同名 `.mr` 禁止）

- `import xxx` 将模块名 `xxx`（不含扩展名）解析为**唯一**物理文件 `xxx.mr`。
- 若在同一项目资源范围内，存在**两条不同路径**均对应同一模块名 `xxx`（例如同时存在 `libs/xxx.mr` 与用户目录下的 `xxx.mr`），必须报错并终止（**不得**采用优先级或覆盖策略择一加载）。

## 6. `.mr` 内容边界

- `.mr` 内函数仅允许“函数头声明 + 附加信息”。
- `.mr` 内允许使用 **`namespace`** 声明命名空间（见第 6.3 节）；**仅 `.mr` 可声明命名空间**。
- `.mr` 内不得出现函数体实现。
- 若检测到函数体实现，必须报错并终止该 `.mr` 文件加载。

## 6.1 `.mr` 的“扫描”与“加载/解析”区分

- 解释器在项目扫描阶段必须发现并记录 `.mr` 资源节点（写入资源表）。
- 资源表中的存在不等价于 `.mr` 被加载/解析。
- `.mr` **仅当被某个 `.obr` 的最外层 `import` 语句显式导入时** 才允许进入加载/解析流程。
- **例外**：`system.mr` 按 [系统头规范](system.md) 隐式导入，不要求源码中出现 `import` 才触发解析。

## 6.2 `.obr` 中的函数定义（`deRfun`）

- 所有 `.obr` 源文件中的**函数定义**必须使用 **`deRfun`** 关键字书写（含函数体）。
- **先声明、后定义（强制）**：在对该 `.obr` 进行**整文件语义检查**（或等价的一遍单元语义分析）时，在接纳某条 `deRfun` **定义**（含函数体）之前，解释器必须已能确定：在**当前 `.obr` 已加载/解析的 `.mr` 集合**中，**存在恰好一条**可与之配对的 `deRfun` **函数头声明**（仅以分号结尾、无函数体）。
  - **声明与定义的匹配**须同时满足：(1) 第 7.8 节**签名判定键**一致（**限定函数名** + 参数个数 + 参数类型序列（按声明顺序））；(2) **返回类型**一致。
  - **已加载/解析的 `.mr` 集合**包括：隐式导入的 [`system.mr`](system.md)，以及该 `.obr` 文件全部最外层 `import` 所触发加载/解析的各 `.mr`（见第 5.2、6.1 节）。
  - 若在该集合中**找不到**满足上述匹配条件的声明，或**返回类型**与任一候选声明不一致，必须**报错并终止**对该 `.obr` 的语义接纳（不得推迟到运行期首次调用时才报错）。
  - 若在该集合内，**两条或以上**来自**不同** `.mr` 的声明与待接纳定义满足**同一**匹配（签名判定键与返回类型均相同），必须**报错并终止**（**跨模块重复声明**）。
  - 项目内其它路径上若存在满足匹配的 `.mr`，但**未被**当前 `.obr` 纳入上述已加载集合，**不得**单独作为“已在某 `.mr` 中声明”的依据。
  - 本节**不排除**第 7.3 节 `@Overwrite` 等规则：即使已找到匹配的函数头声明，该 `.obr` 仍可能在**绑定阶段**因 `@Overwrite`（或其它绑定约束）被拒绝。

## 6.3 命名空间 `namespace` 与限定名 `::`

### 6.3.1 谁可以声明命名空间

- **仅** `.mr` 文件中允许出现 **`namespace`** 关键字用于**声明命名空间**。
- **`.obr` 文件中不得**出现 `namespace ... { ... }` 形式的命名空间声明；若出现，必须视为**语法错误**并终止语义处理。
- `.obr` 中**不得**定义命名空间；仅可通过**限定名**与 **`::`** 解析、引用由已加载 `.mr` 导出的命名空间内符号（见下）。

### 6.3.2 `.mr` 中 `namespace` 语法

- 形式：`namespace <标识符> { /* 体 */ }`
  - `<标识符>` 遵循标识符规则（见 [`scope.md`](scope.md)）。
  - **体**内允许：`namespace` 嵌套、`deRfun` 函数头声明（及第 7 章允许的附加特性）、以及本规范允许出现在 `.mr` 中的其它**仅声明**内容；**不得**出现函数体。
  - **`namespace` 块**以闭合 `}` 结束；闭合 `}` 后**不得**出现分号 `;`（若出现，必须视为**语法错误**）。
- 空体 `namespace test {}` 合法：声明存在命名空间 `test`，其中可无成员。
- **同一 `.mr` 文件顶层**多次出现**同名** `namespace N { ... }`（非嵌套）时，必须视为**同一**命名空间 `N`，各次体内的声明**合并**到同一符号视图；若合并后出现第 7.8 节禁止的重复签名，必须报错。
- **跨 `.mr`（已加载集合）**：对当前 `.obr` 已加载/解析的多个 `.mr` 中，若各自在**文件顶层**出现**同名** `namespace N { ... }`，必须视为**同一**命名空间 `N`，体内声明**合并**到同一符号视图；若合并后出现第 7.8 节禁止的重复签名，必须报错。
- **嵌套同名** `namespace A { namespace A { ... } }`：**不合并**外层与内层 `A`；内层 `A` 为独立命名空间节点，其中 `deRfun foo(...)` 的限定函数名为 `A::A::foo`（由外向内逐段拼接，段数与嵌套深度一致）。
- 在非嵌套重名的**单层**命名空间 `N` 的体中直接声明的 `deRfun foo(...)`，其**限定函数名**为 `N::foo`；嵌套但**段名不同**时按由外向内拼接（例如 `outer::inner::foo`）。

### 6.3.3 限定名与 `::`

- **词法（严格）**：限定名**仅**允许为**一个或多个标识符**用 **`::`** 连接，形如 **`标识符(::标识符)+`**；**不得**出现前导 `::`、尾随 `::`、连续 `::` 或空段。
- **限定名**（合法时）形如：`a::b::c`（至少含一段 `::`）。
- **解析语义**：在表达式、语句及声明中，凡出现 **`限定名`**，解释器必须在**当前 `.obr` 已加载/解析的 `.mr` 集合**（含隐式 `system.mr`）所导出的符号中，按命名空间路径解析目标；**不得**将 `::` 左侧段解释为普通局部/文件作用域变量（命名空间名不是值）。
- **`.obr` 中的用法**：允许使用 **`::`** 书写**限定名**，用于：
  - **函数调用**：例如 `test::xx()`；
  - **`deRfun` 实现**：与 `.mr` 中声明的限定函数名一致，例如 `deRfun test::xx():void { ... }`；
  - 以及本语言在其它专题中允许使用限定名的位置（若有）。
- **凡在 `.mr` 中声明时限定函数名含 `::` 的函数**（即声明位于某 `namespace` 体内，而非文件顶层无命名空间的简单名）：在 `.obr` 中调用**必须**使用与该声明一致的**完整限定名**（例如 `N::foo(...)`、`A::A::foo(...)`）；**不得**使用无前缀的简单名仅写 `foo(...)` 指称该函数（见第 7.9 节）。

### 6.3.4 模块名 `import` 与命名空间名（无歧义）

- `import xx` **仅**表示加载并解析对应的 `xx.mr` 文件；**模块名 `xx` 不作为**限定名里的前缀参与符号解析。
- 限定名 `ms::xx()` 中的 **`ms` 一律解析为命名空间名**（由已加载 `.mr` 中 `namespace ms { ... }` 所声明），与 `import` 目标名是否碰巧同为 `ms` **无关**；二者同名不构成冲突，也不产生额外语义绑定。

### 6.3.5 与可见性、`#LINK` 的关系

- 命名空间内的 `deRfun` 声明在**可见性**与 **`#LINK` / 语言空间**规则上，与**未**位于 `namespace` 内的文件级 `deRfun` 声明**适用同一套规则**；仅符号的**限定名**形态不同（见 [`visibility.md`](visibility.md)）。

### 6.3.6 与第 6.2、7 章的关系

- 声明与实现配对、签名判定键、重载与绑定规则中的「函数名」均指 **§7.1 所定义的限定函数名**（可为简单标识符，或 `::` 连接的限定名）；**同一限定函数名 + 参数 + 返回类型** 在配对语义中与无命名空间时一致。

## 7. `deRfun` 函数头声明规范

### 7.1 声明格式

`.mr` 文件中的函数头声明关键字为 `deRfun`，格式如下：

```text
[特性A]
[特性B]
deRfun [限定函数名]([类型]参数,[类型]参数...):[返回值];
```

说明：

- **限定函数名**：为**简单标识符**（例如 `WriteBytes`、`main`），或 **`::` 连接的限定名**（例如 `test::xx`），与第 6.3 节命名空间规则一致；用于重载与配对时，**整条**限定名参与「函数名」比较，而非仅最后一段。
- `deRfun` 声明必须以分号 `;` 结束。
- `[]` 仅表示语法占位，不是字面量。
- 参数列表可为空；非空参数以逗号分隔。

### 7.2 特性关键字

- 函数头特性通过 `@Overwrite(...)` 表达。
- 不写 `@Overwrite` 时，默认等价于 `@Overwrite()`。
- 每个函数头声明仅允许存在一个 `@Overwrite` 特性。
- 若同一函数头出现两个或以上 `@Overwrite`，必须报错。
- 函数头可另附 **`@Callfun(...)`** 特性，用于约束“哪些 `.obr` 允许调用该函数头所绑定的实现”（见第 7.4 节）；每个函数头声明仅允许存在一个 `@Callfun`；若出现两个或以上 `@Callfun`，必须报错。

### 7.3 `@Overwrite` 语义

`@Overwrite` 用于定义“哪些 `.obr` 文件允许或禁止实现该函数头”。

- `@Overwrite()`：允许任意 `.obr` 实现。
- `@Overwrite(a.obr,b.obr)`：仅允许列出的 `.obr` 实现。
- `@Overwrite(!a.obr,!b.obr)`：禁止列出的 `.obr` 实现。
- `@Overwrite(lib/,util/)`：允许列出目录下的 `.obr` 实现。
- `@Overwrite(lib/,util/,!util/a.obr)`：允许 `lib/` 与 `util/`，但显式排除 `util/a.obr`。

#### 7.3.1 通配符 `*`

- `@Overwrite` 的路径项中允许使用 **`*`** 作为通配符。
- `*` 表示对路径的**通配**，可匹配**零个或多个路径段**（可跨越 `/`），在规范化逻辑路径上按子路径模式匹配；同一项中可出现多个 `*`。
- 通配符匹配必须在同一套规范化路径空间内完成（与 [`#LINK`](preprocessor.md) 的路径项一致）；若匹配结果导致同一函数头在绑定阶段出现多个无法消歧的实现，必须报错。

### 7.4 `@Callfun` 语义（调用约束）

`@Callfun` 用于定义“哪些 `.obr` 文件允许**调用**该函数头已绑定的实现”（与 `@Overwrite` 的实现约束不同：前者管**调用**，后者管**实现**）。

- 不写 `@Callfun` 时，默认等价于 `@Callfun(*)`：允许任意 `.obr` 调用（在语言空间与可见性规则允许的前提下）。
- `@Callfun(a.obr,b.obr)`：仅允许列出的 `.obr` 发起调用。
- `@Callfun(!*)`：**禁止一切调用**（包括来自其它 `.obr` 的调用）。此时函数头仍允许被实现（由 `@Overwrite` 决定），但除**解释器规定的入口/bootstrap 调用**外，不得从用户代码路径发起调用。

**入口函数 `main` 的例外**：

- 对 `deRfun main():void`：解释器在启动程序时**必须**调用 `main.obr` 中的实现；该调用**不受** `@Callfun(!*)` 限制；其它 `.obr` 对用户代码是否可调用 `main` 仍受 `@Callfun` 约束。

### 7.5 组合规则

- `@Overwrite` 参数列表允许“允许项”和“禁止项”混合出现。
- 禁止项使用前缀 `!` 表示。
- 当同一目标同时命中允许项与禁止项时，禁止项优先。

### 7.6 换行规则

- `@Overwrite` 与 `@Callfun` 参数中的逗号允许换行。
- 判定规则与 `#LINK` 一致：
  - 逗号后若存在除空格字符外的其他字符，则当前行结束；
  - 逗号后若不存在除空格字符外的其他字符，则下一行作为同一参数串续写。

### 7.7 示例

```text
@Overwrite(util/io.obr)
deRfun WriteBytes([int]baseMemory,[int]offsetByte):boolean;
```

命名空间与限定名（声明于 `.mr`；在 `.obr` 中实现与调用时使用 `test::` 前缀）：

```text
namespace test {
    deRfun xx():void;
}
```

### 7.8 函数重载与签名规则

- ObjectiveR 支持函数重载。
- 函数签名必须按参数信息确定，不按返回值区分重载。
- **签名判定键**为：**限定函数名**（见第 7.1 节）** + 参数个数 + 参数类型序列（按声明顺序）**。
- 仅返回值不同、参数信息相同的多个 `deRfun` 声明必须视为签名冲突并报错。
- 同一 `.mr` 文件内若出现两条或以上「签名判定键 + 返回类型」均相同的 `deRfun` 声明，必须报错。
- 同名函数只要参数类型序列不同，即视为不同重载签名。
- **用途区分**：上述「签名判定键」用于**重载分辨与调用匹配**（见第 7.9 节）。**`.mr` 函数头与 `.obr` 函数实现的配对**除满足该键外，还须**返回类型**一致（见第 6.2、7.10 节）；不得仅凭参数部分一致而忽略返回类型。

### 7.9 重载调用匹配规则

- 调用函数时，解释器必须按“**限定函数名** + 实参数量”先筛选候选函数头：**简单名**调用（无前缀 `::`）仅匹配**未**置于 `namespace` 内声明的函数头（即 `.mr` 中为简单限定函数名的 `deRfun`）；**带 `::` 的调用**使用完整限定名，匹配对应命名空间内的声明（见第 6.3.3 节：仅存在于命名空间内的函数**不得**用简单名调用）。
- 在候选集合中，按“实参类型序列与形参类型序列完全一致”进行匹配。
- 若存在且仅存在一个完全匹配项，则该项为唯一命中目标。
- 若不存在完全匹配项，必须报错。
- 若出现多个完全匹配项，必须报错（重载二义性）。

### 7.10 绑定约束

- `.obr` 中的函数实现与 `.mr` 函数头绑定时，必须使用同一签名判定键，且**返回类型**一致。
- 接纳 `.obr` 中的 `deRfun` 定义之前，必须先满足第 6.2 节“先声明、后定义”：在当前 `.obr` 的已加载 `.mr` 集合中，须能**唯一**确定与之配对的一条函数头声明（见第 6.2 节跨模块重复声明规则）。
- 任一实现只能绑定一个同签名函数头。
- 任一函数头若被多个实现同时声明为对应目标，必须报错。

### 7.11 函数未实现错误

- 当 `.obr` 代码执行一次函数调用时，若该调用在可见性与语言空间规则下解析到某个来自 `.mr` 的 `deRfun` 函数头（且签名已确定），则必须保证该 `deRfun` 函数头存在至少一个绑定的 `.obr` 函数实现。
- 若该 `deRfun` 函数头对应的绑定实现集合为空，则必须报错，错误类型/错误信息为：`函数未实现（Function Not Implemented）`。

### 7.12 调用被禁止错误（`@Callfun`）

- 当某次调用在绑定上命中了某个 `deRfun` 函数头，但该函数头的 `@Callfun` 规则禁止当前调用方（例如 `@Callfun(!*)` 禁止除解释器入口外的调用），则必须报错（调用不允许）。

### 7.13 系统入口 `main()`（`system.mr` 与 `main.obr`）

[`system.mr`](system.md) 中声明程序入口函数头，典型形式如下（`@Overwrite` 路径中允许使用通配符 `*`）：

```text
@Callfun(!*)
@Overwrite(*/main.obr)
deRfun main():void;
```

语义要点：

- **`main()`** 签名为无参数，返回类型为 `void`（见 [`types.md`](types.md) 中 `void`）。
- `@Callfun(!*)` 表示禁止一般用户代码路径调用 `main`；**解释器启动程序时对 `main` 的调用**不受该限制（见第 7.4 节）。
- 文件名为 **`main.obr`** 的源文件**必须**提供与上述函数头签名一致的实现，并作为解释器唯一程序入口实现；实现体使用 `deRfun` 关键字并带函数体，例如：

```text
deRfun main():void{
    // pass
}
```

- 若 `main.obr` 中不存在可识别的 `deRfun main():void` 实现，必须报错并终止（与 [执行模型](runtime.md) 一致）。

- 与 `main` 并列的其它系统头声明（例如 **`namespace std`** 与 **`std::rout`**）见 [`system.md` 第 1.1 节](system.md)。

## 8. 导入绑定与 `.mr` 状态共享

### 8.1 导入绑定关系

- 对任一 `import xxx` 语句，记导入方为 `O`（某个 `.obr` 文件），被导入模块为 `X`（某个 `.mr` 文件）。
- 在语义上，`O` 与 `X` 建立导入绑定关系。
- 该导入绑定关系用于决定：`O` 可访问由 `X` 声明或标识的函数与宏。

### 8.2 加载/解析唯一性

- 同一个 `.mr` 文件 `X` 在解释器实例内的加载/解析结果必须唯一。
- 因此，不同 `.obr` 文件对同一 `X` 的导入（显式 `import` 或对 `system` 的隐式导入）必须复用同一份加载/解析结果，而不得生成多份彼此独立的 `.mr` 语义数据。

### 8.3 状态共享

- “共享 `.mr` 状态”指：复用的加载/解析结果对所有导入方共同生效。
- 即：不同 `.obr` 导入同一个 `.mr` 时，使用同一份 `.mr` 声明/宏等语义数据，并共享由该 `.mr` 导入所产生的语义效果。

### 8.4 生命周期：冻结与恢复

- 对任一 `.mr` 文件 `X`，在解释器实例内存在唯一的 `X` 实例语义数据（来自 `.mr` 的解析结果与相关模块化声明）。
- 记 `ActiveExecutionObrs(X)` 为 `.obr` 文件集合，满足：
  - 该 `.obr` 文件已建立对 `X` 的导入绑定（最外层显式 `import X`，或对 `system.mr` 的隐式导入等价于已导入）；
  - 且解释器当前正在执行该 `.obr` 文件中的任何代码（存在活动执行上下文/调用帧属于该 `.obr`）。

生命周期规则（强制）：

1. **执行激活与实例复用**
   - 当某个 `.obr` `O` 因为被调用（例如 `main.obr` 调用 `b.obr` 的函数）而进入活动执行状态时，`O` 的所有最外层 `import X` 语句都必须使 `X` 进入可用状态。
   - 若这是 `X` 的首次被任何活动执行 `.obr` 触发，`X` 必须在此时解析并完成实例语义数据构建。
   - 若 `X` 已解析并处于可用或冻结状态，后续活动执行仅复用同一份 `X` 实例语义数据，不得重复初始化或创建新实例语义数据。

2. **.obr 退出（释放条件）**
   - 当且仅当解释器不再执行任何属于某个 `.obr` `O` 的代码（即该 `.obr` 已“退出”，不再存在活动执行上下文/调用帧属于 `O`）时：
     - 从 `ActiveExecutionObrs(X)` 中移除 `O`。
   - 当 `ActiveExecutionObrs(X)` 变为非活动（为空）时，触发冻结等待。

3. **冻结对象与冻结行为**
   - 冻结只针对与 `X` 相关且已建立绑定的“函数实现绑定/实现态”，并且冻结包含其实现函数体在运行期间产生/持有的 `static` 值与静态状态。
   - 即：只要某个 `deRfun` 函数头在作用空间内存在被 `X` 接管的具体 `.obr` 实现（即“函数头被实现”），则该实现函数体对应的静态状态在冻结时必须被冻结并保持。
   - 冻结不重置 `X` 的任何绑定结果；冻结后该绑定结果及其静态状态应保持在冻结态，直到恢复触发。
   - 在本规范条件下，冻结不会操作 `.mr` 自身声明内容（因为 `.mr` 内仅存在函数头与注解，不含函数体实现）。

4. **恢复（非重置）**
   - 若之后出现新的 `.obr` 在活动执行中再次触发 `import X`，且 `X` 处于冻结态，则必须恢复该冻结的“函数实现绑定/实现态”继续使用。
   - 恢复必须复用冻结态的数据，禁止重置为冻结前的初始状态。

### 8.5 关于“作用空间唯一实例”的一致性约束

- 若一个 `.mr` 的语义被标识/归属到某个作用空间，则该作用空间内应当只存在该 `.mr` 的一个实例语义数据源。
- 因此，第 8.4 条的复用、冻结与恢复都必须以“作用空间内唯一实例语义数据”作为前提。

## 9. `.mr` 版本兼容与预处理指令约束

### 9.1 版本兼容性

- `.mr` 之间彼此兼容。
- 高版本 Obr 解释器应当能够处理并解析低版本 `.mr`。

### 9.2 `.mr` 禁止预处理指令与标识

- `.mr` 文件中不得出现任何预处理指令与标识（例如不应出现以 `#` 开头的行，且不得包含 `#VERSION`、`#LINK` 等任何预处理指令）。
- `.mr` 不需要也不允许携带语言版本标识。

### 9.3 解析失败的错误类型

- 若某个 `.mr` 文件在当前解释器中无法解析，则必须统一报“语法错误”（Syntax Error），不得使用与版本不兼容或其它类别相关的错误类型来替代。
